## 一、這堂課真正要學的是什麼？

這堂課不是單純列出熱門 AI 工具，而是把工作分成四個層次：

1. **文字與知識整理**：把零散來源收進專案、Gem 或 Notebook，形成可持續累積的知識庫。
2. **圖像生成與後製**：模型先產生方向，再依需求選擇「請 AI 修改」或「拆成圖層人工精修」。
3. **AI Agent**：AI 不只回答問題，而是讀取專案、使用工具、執行命令並完成多步任務。
4. **流程自動化**：把已經驗證可行的工作流程交給觸發器定期或依事件執行。

### 生活化比喻

把 AI 工作方式想成一間新的辦公室：

- 文字工具是負責整理資料與寫初稿的同事。
- 生圖工具是視覺設計與攝影同事。
- AI Agent 是可以走進工作現場、操作工具並交付成果的專案助理。
- 自動化程式則是每天準時執行固定工作的行政系統。

重點不是每一件事都交給 AI，而是知道任務適合交給哪一層，以及在哪裡需要人工檢查。

## 二、文字篇：從聊天走向可累積的知識系統

### 2.1 專案、Gem、Notebooks 與 Markdown（投影片 3）

![專案、Gem、Notebooks 與 Markdown](slides/slide-3.png)

#### 白話解釋

- **專案（Project）**像一間專用工作室，同一主題的資料、對話與成果都放在一起。
- **Gem**像事先受過訓練的固定角色，會依你設定的身份、語氣、規則與任務回答。
- **Notebook**像分類清楚的資料櫃，用來保存來源、筆記與主題脈絡。
- **Markdown（.md）**像跨平台通用的資料標籤，檔案輕、結構清楚，也不容易被單一平台綁住。

#### 正式定義

Markdown 是用純文字符號表示標題、清單、連結與程式碼等結構的輕量標記語言。它能被 Obsidian、GitHub、許多 AI 工具與文件轉換系統讀取，因此適合長期保存及跨工具流通。

#### 老師示範所呈現的工作方式

錄影前段以「颱風預測專家」為例，建立包含台灣情境的角色指令。指令不只說「你是颱風專家」，而是進一步交代：

- 熟悉的專業領域，如路徑、強度、海溫、風切、副熱帶高壓與地形效應。
- 需要回答的實際問題，如各地風雨、停班停課、交通、航班、沿海與山區安全。
- 回答風格要專業、清楚、務實，而且符合台灣使用情境。

![錄影中的颱風專家角色指令](video-frames/frame-02.jpg)

這個案例說明：**角色指令的品質取決於任務邊界、專業範圍、在地脈絡與輸出要求，而不是角色名稱寫得多華麗。**

#### 建議操作流程

1. 先建立一個明確主題的專案。
2. 把「來源資料」與「自己的整理」分開保存。
3. 撰寫角色指令，包含身份、任務、限制、對象與輸出格式。
4. 將穩定可重用的成果另存為 `.md`。
5. 換工具前先確認 Markdown 檔案仍保留標題層級、連結與來源。

### 2.2 Gemini × NotebookLM（投影片 4）

![Gemini 與 NotebookLM 的分工](slides/slide-4.png)

#### 白話解釋

Gemini 像會陪你討論、發想和寫作的研究助理；NotebookLM 像守著指定資料、能指回原始來源的圖書館員。

#### 兩者的分工

- **Gemini**：對話、發想、生成內容、改寫與延伸。
- **NotebookLM**：整理資料、摘要、依來源問答、建立可追溯的知識脈絡。
- **Gemini Notebooks**：把互動生成與 NotebookLM 式的來源整理放在同一個工作脈絡中。

適合情境包括主題研究、教學備課、專案資料整理、菜單設計或把多份來源逐步整理成個人知識庫。

#### 使用原則

1. 需要可查證結論時，先把可信來源放進 NotebookLM。
2. 需要發想、改寫與內容延伸時，再讓 Gemini 協助。
3. 生成內容不等於來源內容；重要結論仍要回到原始資料核對。

### 2.3 直接產生 DOCX、Excel 等文件（投影片 5）

![GPT 與 Gemini 的文件輸出](slides/slide-5.png)

聊天回答只是中間結果。真正能進入工作流程的成果通常是：

- Word／DOCX：會議紀錄、教案、報告、通知、長文與正式文件。
- Excel／試算表：清單、欄位資料、公式、彙整表、分析與圖表。
- Markdown：需要跨平台保存、版本管理或後續再加工的知識內容。

#### 交付前檢查

- 檔案能否正常開啟。
- 標題、段落、表格與欄位是否正確。
- Excel 公式與數字是否合理。
- 引用與資料來源是否可查證。
- 格式是否方便下一位使用者繼續編輯。

## 三、生圖篇：生成只是第一步，修正方法決定成品品質

### 3.1 生圖模型的能力取向（投影片 7）

![生圖模型比較](slides/slide-7.png)

簡報以 Nano Banana 與 ChatGPT Image 為例，提醒我們比較模型時不能只問「哪一個最好」，而要問：

- 是否需要寫實、高保真的細節？
- 圖片中的中文字是否必須準確？
- 是否需要理解參考圖與複雜語意？
- 是否重視創意構圖、風格變化或角色一致性？

簡報上對模型名稱與版本的標示屬於本次課程內容；日後使用時仍應查看工具當下的實際版本與功能。

### 3.2 生圖修正的兩條路（投影片 8–9）

![生圖修正兩派](slides/slide-8.png)

#### 路線一：請 AI 直接修改

適合：

- 快速改變背景、顏色或局部物件。
- 仍在探索方向，不需要精準完稿。
- 希望以自然語言快速迭代。

限制：局部重畫有時會連帶改變人物、構圖、字體或其他原本正確的細節。

#### 路線二：拆成圖層自行修改

適合：

- 文字、位置與品牌色必須精準。
- 需要把人物、背景、裝飾與文字分開調整。
- 圖片要進入 Canva 等工具完成正式交付。

課堂示例包含：先讓影像工具產生去背 PNG 或細節圖，再使用 Canva 的圖層／魔法圖層功能拆解元素。

![去背與 Canva 圖層示例](slides/slide-9.png)

#### 實務判斷

- **先求效率**：請 AI 修。
- **需要精修完稿**：拆成圖層自己修。
- 最穩定的流程通常是「AI 產生方向 → 選定版本 → 圖層化 → 人工完成最後 10%」。

## 四、AI Agent 篇：從回答問題升級為執行任務

### 4.1 什麼是 AI Agent？

#### 白話解釋

一般聊天 AI 像提供建議的顧問；AI Agent 像能進入工作環境的數位同事，它會讀檔案、使用工具、執行命令、檢查結果，並持續完成多個步驟。

#### 正式定義

AI Agent 是能根據目標感知環境、規劃步驟、調用工具、執行動作並依結果調整的軟體系統。它仍需要清楚的權限範圍、驗收條件與人工監督。

### 4.2 OpenCode（投影片 11）

![OpenCode](slides/slide-11.png)

課堂列出的能力包括：

- 自動生成與修改程式碼。
- 快速讀取專案結構。
- 整合終端機指令與流程。
- 協助除錯及提出優化建議。
- 支援不同模型，保留整合彈性。

### 4.3 Codex（投影片 12）

![Codex](slides/slide-12.png)

Codex 的課堂定位是程式開發 Agent，可理解專案、支援多種程式語言、執行終端命令、整合 Git 版本控制並處理多步自動化任務。

#### 好的任務交付方式

1. 說清楚目標，而不只說「幫我改好」。
2. 指出不能改動的範圍。
3. 提供驗收方式，例如測試、輸出格式或範例。
4. 先讓 Agent 讀取真實檔案與目前狀態。
5. 完成後檢查檔案差異與測試結果。

### 4.4 Claude Code（投影片 13）

![Claude Code](slides/slide-13.png)

課堂強調的能力包括自然語言開發、專案脈絡理解、程式生成、重構、除錯、測試與開發工具整合。它的價值不是單次產生程式碼，而是能理解既有程式庫後，在原有架構中進行修改。

### 4.5 Antigravity 與多 Agent 協作（投影片 14）

![Antigravity](slides/slide-14.png)

課程用 Antigravity 說明任務編排與多 Agent 協作：

- 把大目標拆成可執行步驟。
- 讓不同 Agent 依專長分工。
- 串接工具與外部資源。
- 將重複步驟自動化。
- 保留整體專案脈絡與交付方向。

多 Agent 並不是「Agent 越多越好」。每一個角色都必須有明確輸入、輸出、交接條件與失敗處理，否則只會增加溝通成本。

### 4.6 AI 與微軟文書工具（投影片 15）

![AI 與微軟文書工具](slides/slide-15.png)

- **PPT**：產生大綱、投影片內容、版面與視覺建議。
- **Excel**：整理與清理資料、解釋或生成公式、製作圖表及報表。
- **Word**：撰寫與編輯文件、格式排版、摘要長文。

課堂比較中，ChatGPT 偏向快速生成、創意與分析；Claude 偏向長文理解、內容結構與多文件格式。實際使用時仍應以任務、資料與交付品質決定工具。

### 4.7 連結 NotebookLM 的兩條途徑（投影片 16）

![NotebookLM 連結途徑](slides/slide-16.png)

#### 途徑一：Gemini 直接連用 NotebookLM

- 上手快、操作直覺。
- 適合日常摘要、問答與知識整理。
- 比較容易維持來源脈絡。

#### 途徑二：AI Agent 串接 notebooklm-mcp-cli

- 能把 NotebookLM 放進更長的工具流程。
- 適合進階任務、流程整合與重複工作。
- 需要處理登入授權、工具權限與連線失敗。

錄影後段實際出現 Google 登入／授權未完成的情況。這是一個重要示範：**MCP 串接不是只安裝工具就完成，還要確認帳號、授權、可存取範圍與失敗時的回復方式。**

## 五、自動化篇：Google Apps Script（投影片 18）

![Google Apps Script](slides/slide-18.png)

> **錄影範圍提醒**：老師在約 `00:57:02` 明確表示時間不足，「剩下自動化講不完」，因此錄影沒有完整講授 GAS。本節是依投影片 18 整理的延伸筆記，不能當成老師已在影片中逐項示範。

### 白話解釋

GAS 像 Google Workspace 裡的小管家。到了指定時間、有人送出表單或試算表被修改時，它就依規則寄信、整理資料、建立文件或更新報表。

### 正式定義

Google Apps Script 是 Google 提供、以 JavaScript 為基礎的雲端腳本平台，可操作 Sheets、Docs、Gmail、Drive、Forms、Calendar、Slides 等服務。

### 五個核心觀念

1. **串接 Google 生態系**：同一段程式可在多項服務之間搬運與處理資料。
2. **流程自動化**：減少人工複製貼上與重複操作。
3. **觸發器**：依時間、編輯、表單提交或自訂事件啟動。
4. **JavaScript 基礎**：語法相對普及，並有完整 API 文件與社群資源。
5. **應用可逐步擴大**：從自動寄信、資料整理，到報表、系統整合與專屬工具。

### 建議的第一個 GAS 專案

選擇一個風險低、規則固定的任務，例如：表單送出後，把資料寫進試算表並寄出確認信。

實作前先寫清楚：

- 什麼事件會啟動流程？
- 需要讀取哪些資料？
- 會修改或寄出什麼內容？
- 發生錯誤要記錄在哪裡？
- 如何用測試資料確認沒有重複寄信或覆蓋正式資料？

## 六、四大技能如何串成一條工作流？

以「製作一份課程學習網頁」為例：

1. **文字層**：把講義、錄影逐字稿與自己的理解整理成 Markdown。
2. **生圖層**：擷取簡報或產生輔助視覺，再用圖層工具精修。
3. **Agent 層**：讓 Agent 讀取素材、建立網站、測試按鈕與修正錯誤。
4. **自動化層**：未來新增課程資料時，自動建立資料夾、轉錄、更新索引或寄送提醒。

### 最重要的順序

**先把一次流程做對，再把它交給 Agent；先確認 Agent 的結果穩定，再考慮自動化。**
